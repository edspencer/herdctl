## Key Findings

### 🆕 Finding #012: Web API Lacks Authentication (NEW - HIGH)

**Severity:** HIGH
**Status:** OPEN
**Discovery:** 2026-03-06 audit

**Description:**
The web dashboard added four new REST API endpoints in commit 01274a8 (PR #144) with no authentication:
- `GET /api/chat/recent` - List recent sessions across all agents
- `GET /api/chat/config` - Read fleet web config
- `GET /api/chat/all` - List all discovered sessions
- `GET /api/chat/session/:encodedPath/:sessionId` - Read arbitrary session files

**Security Impact:**
- **Information disclosure:** Full access to all chat session history
- **Session enumeration:** Attacker can list all sessions and working directories
- **Credential exposure:** Sessions may contain OAuth tokens (Finding #011)
- **No audit trail:** No logging of who accessed what sessions

**Risk Assessment:**
- **Localhost deployment (default):** LOW risk - implicit trust boundary
- **LAN/Internet deployment:** HIGH risk - unauthorized access to sensitive data

**Evidence:**
- Routes registered with no authentication middleware (chat.ts:22-27)
- No JWT, session cookies, API keys, or authorization headers present
- CORS allows localhost origins only (consistent with local dev use)

**Recommendation:**
1. **Document** that web dashboard is localhost-only by design
2. **Warn** against binding to `0.0.0.0` or exposing on network
3. If network deployment needed: implement JWT/session auth, path validation, audit logging

**Related:**
- Finding #011 (OAuth credentials in sessions)
- Q1 (webhook authentication)

**Severity:** MEDIUM → MEDIUM (risk elevated)
**Status:** OPEN (unchanged)

**Update:** 2026-03-06

**New Risk Vector:**

- Debug output from container-manager.ts OAuth functions

**Combined Impact:**

**Recommendation:**

2. Review `jsonl-parser.ts` to ensure it doesn't expose sensitive fields
3. Add credential redaction to session export/API functions
4. Implement file permissions enforcement (0600) for `credentials.json`

## New Attack Surface Analysis

### 1. Agent Distribution System (Commit bd59195, PR #99)

**New Package:** `packages/core/src/distribution/`

**Entry Points:**
- `herdctl agent add github:owner/repo` - Clone from GitHub
- `herdctl agent add ./local/path` - Install from local directory

**Attack Vectors Analyzed:**

#### ✅ Agent Name Validation - SAFE
**Status:** Properly validated before file operations

**Validation Chain:**
1. Schema validation: `AgentConfigSchema` enforces `AGENT_NAME_PATTERN`
2. Explicit validation: `validateAgentName()` checks pattern before path operations
3. Pattern: `/^[a-zA-Z0-9][a-zA-Z0-9_-]*$/` prevents traversal

**Pattern Security Properties:**
- ✓ Prevents `../` (no dots, no slashes)
- ✓ Prevents absolute paths (must start with alphanumeric)
- ✓ Prevents Windows traversal `..\\` (no backslashes)
- ✓ Prevents hidden files (must start with alphanumeric, not `.`)

**Assessment:** No vulnerability. Validation occurs before all `path.join()` operations.

#### ⚠️ GitHub SSRF Potential - MEDIUM RISK
**Status:** User controls GitHub URLs with no allowlist

**Implementation:** `repository-fetcher.ts` clones from any GitHub URL:
```typescript
git clone https://github.com/${owner}/${repo}.git
```

**Mitigations Present:**
- 2-minute timeout
- `GIT_TERMINAL_PROMPT=0` prevents interactive prompts
- Shallow clone only (`--depth 1`)

**Remaining Risk:**
- No URL validation/allowlist
- Could be used for reconnaissance (valid/invalid repo detection)
- Git may use cached credentials if present

**Recommendation:** MEDIUM priority - Add GitHub URL validation or org/repo allowlist if strict security needed. Current risk is low for trusted users.

**Related:** Q11 (GitHub SSRF in work sources)

#### ✅ Local Path Handling - ACCEPTABLE
**Status:** User controls local paths, which is expected CLI behavior

**Implementation:** Copies local directories to temp directory for validation.

**Security Considerations:**
- User explicitly specifies local path
- Path is validated for existence and directory type
- Copies entire directory including potentially sensitive files (.git, .env)

**Assessment:** Acceptable - user has explicit control over what they install. This is standard package manager behavior.

### 2. Session Discovery System (Commit 01274a8, PR #144)

**New Package:** `packages/core/src/state/session-*.ts`

**Entry Points:**
- Web API routes (covered in Finding #012)
- File system scanning: `~/.claude/projects/`, `.herdctl/docker-sessions/`

**Attack Vectors Analyzed:**

#### ⚠️ encodedPath Validation - PARTIAL PROTECTION
**Status:** Indirect validation via groups lookup

**Implementation:**
- Route accepts `encodedPath` parameter
- Looks up `encodedPath` in pre-computed session groups
- Returns 404 if not found
- Uses group's `workingDirectory` for file operations

**Security Analysis:**
The route doesn't validate `encodedPath` directly but constrains it to the list of working directories discovered by `SessionDiscoveryService`. This provides protection but relies on:
1. Discovery service only finding legitimate directories
2. No bugs in path encoding/decoding logic
3. No race conditions in group computation

**Encoding/Decoding Logic:**
```typescript
// Encoding: Replace path separators with hyphens
encodedPath: session.workingDirectory.replace(/[/\\]/g, "-")

// Decoding: Reconstruct path from hyphenated string
function decodePathForDisplay(encodedPath: string): string {
  if (encodedPath.startsWith("-")) return "/" + encodedPath.slice(1).replace(/-/g, "/");
  if (/^[A-Za-z]:-/.test(encodedPath)) return encodedPath[0] + ":" + encodedPath.slice(2).replace(/-/g, "/");
  return encodedPath.replace(/-/g, "/");
}
```

**Assessment:** MEDIUM risk - Should add explicit validation of `encodedPath` against malicious patterns (e.g., regex check for valid encoded format) before using it, even though the groups lookup provides indirect protection.

**Recommendation:**
```typescript
// Add explicit validation
if (!/^[A-Za-z0-9:-]+$/.test(encodedPath)) {
  return reply.status(400).send({ error: "Invalid path encoding" });
}
```

#### ✅ JSONL Parsing - SAFE

**Implementation:** `jsonl-parser.ts` parses session files line-by-line with per-line try/catch.

**Security Properties:**
- Malformed JSON lines are skipped
- Parser doesn't crash on invalid input
- No code execution risk (JSON parsing only)

**Assessment:** Safe implementation for untrusted input parsing.
