## Process Failures

The `/security-audit` command is optimized for incremental change detection but provides **zero value on zero-change days**. This creates several problems:

1. **Missed Investigation Opportunities**: When no code changes exist, audit capacity should shift to:
   - Answering open questions
   - Triaging overdue findings
   - Performing staleness checks on critical hot spots
   - Proactive threat modeling

2. **Overdue Findings Accumulate**: Finding #013 is 16 days overdue with no intervention mechanism. The audit flags it as "IMMEDIATE PRIORITY" but takes no action.

3. **Open Questions Stagnate**: 7 medium-priority questions (Q1, Q4, Q5, Q7, Q8, Q13, Q15) have been open for weeks/months. No systematic investigation plan exists.

4. **Dependency Drift**: npm vulnerabilities remain static. No automated remediation or investigation occurs.

**Root cause**: The audit process lacks a "maintenance mode" for zero-change scenarios.

## Proposed Process Improvements

### For `/security-audit` Command

Add a **Phase 2a: Zero-Change Mode** that activates when commits_since_audit == 0:

```markdown
**Phase 2a: Zero-Change Maintenance (NEW)**

IF commits_since_audit == 0:
  - Check for overdue findings in FINDINGS-INDEX.md
  - IF overdue critical/high findings exist:
    - Spawn finding-investigator to triage
  - ELSE IF high-priority open questions exist:
    - Spawn question-investigator (investigate 1-2 questions)
  - ELSE:
    - Perform staleness check on critical hot spots
    - Run creative attack ideation session

This ensures every audit provides value, even with zero code changes.
```
