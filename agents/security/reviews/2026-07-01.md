## Gaps Identified

### 1. Hot Spots Coverage - Incomplete

**Finding**: Phase 3 states "NOT REQUIRED" because only path-safety.ts and cli-session-path.ts changed. However, HOT-SPOTS.md specifies that **Critical Hot Spots (Always Check)** should be verified every audit, regardless of whether they changed.

**Evidence**:
- HOT-SPOTS.md Section: "Critical Hot Spots (Always Check)" states these "must be reviewed every audit"
- The audit skipped verification of:
  - `container-manager.ts` - OAuth credential handling, logger leak checks
  - `container-runner.ts` - Shell escaping completeness
  - `schema.ts` - New string fields without validation patterns
  - `interpolate.ts` - Still only ${VAR} substitution
  - `shell.ts` - Timeout and output limits

**Impact**: Medium - These files could have introduced regressions without notice

**Recommendation**: Always verify the checklist in HOT-SPOTS.md, even if files haven't changed. Scanner results alone are insufficient.

### 3. Discord Connector - Out of Scope Too Quickly

**Finding**: Discord connector changes (commit 3f947a0) were marked "OUT OF SCOPE" without basic security checks.

**What Was Missed**:
- Discord bot exposes fleet control to external users via network
- No verification of authentication/authorization patterns
- No check for command injection via Discord message content
- No verification that Discord commands validate agent names before execution
- Could be a major new attack surface

**Actual Risk**: The audit correctly notes this needs review but punts it entirely. A basic sanity check would be appropriate:
- Are Discord message payloads validated before use?
- Are agent names from Discord sanitized?
- Is there rate limiting?

**Impact**: Medium - New external attack surface with unknown security posture

**Recommendation**: "Separate review needed" is fine, but perform a 15-minute sanity check first. Add Discord security review to backlog with specific questions.

### 4. Attack Ideation - Formulaic

**Finding**: Investigation focuses on obvious risks (path traversal, shell injection) but doesn't explore novel attack combinations specific to herdctl's architecture.

**Examples of Unexplored Scenarios**:

- Can malicious agent output in job files cause issues when read by web dashboard?
- Can rapid programmatic agent creation/deletion cause state directory race conditions?
- Does FleetManager.getAgentSessions() expose sessions from other users in multi-user scenarios?

**Impact**: Low - These may all be non-issues, but creative thinking wasn't demonstrated

**Recommendation**: Add "Attack Scenario Brainstorming" step to audit process. Spend 10 minutes thinking creatively about architecture-specific risks.
